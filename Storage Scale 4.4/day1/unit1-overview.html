What Storage Scale is (and isn’t)

Storage Scale (formerly GPFS) is a clustered file system designed for concurrent access from multiple nodes, scaling capacity and performance while keeping a single namespace.

Key mental model: Storage Scale is not a classic “single filer”. There is no single metadata server bottleneck, and the system relies on quorum to prevent split-brain.
Core components you must name without hesitation
  • Nodes (clients, NSD servers, quorum nodes)
  • NSDs (Network Shared Disks) exporting storage to the cluster
  • Daemon + kernel extension (I/O path + VFS integration)
  • Pagepool (Storage Scale cache, not OS page cache)
  • File system descriptor (replicated metadata on disks)
Exam trap: “Quorum improves availability” (Nope.)

Quorum primarily protects integrity by preventing multiple partitions from writing independently. The system would rather stop than corrupt.

Why correct: Any statement that frames quorum as integrity protection matches design intent.
Why wrong: Statements implying “quorum keeps the cluster running at all costs” are backwards: it may halt I/O to prevent split-brain.

Diagram — Parallel I/O (client → NSD servers → disks)

The important bit: clients can drive parallel I/O across multiple NSDs/servers; no single choke point by design.

Why correct: “Storage Scale stripes and distributes I/O across NSDs/servers, avoiding a single bottleneck.”
Why wrong: “Storage Scale depends on one metadata server” (that’s a different design pattern).

Diagram — Quorum prevents split-brain

When the network partitions, two groups might both believe they should continue. Quorum rules prevent concurrent writers.

Operational implication: if quorum is lost, the safest behavior is to stop one side. That’s data integrity doing its job.
Why correct: “Quorum prevents split-brain by allowing only one partition to continue I/O.”
Why wrong: “Both sides can keep writing and reconcile later” (that’s fantasy unless your system is built for it).
