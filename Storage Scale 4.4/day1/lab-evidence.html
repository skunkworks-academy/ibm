Submission format

Create one text file per node (example: node01_day1.txt). Paste outputs exactly as shown below. Include the command and the output.

0%
0/0 checkpoints complete
If you can’t provide these outputs, you haven’t proven the system state. Troubleshooting starts here.

A) Node identity & OS baseline (each node)

# Paste outputs for each node
hostname -f
cat /etc/redhat-release
uname -r
ip -br a
Why this matters: mismatched identity and kernel versions are the root of most “mysterious” failures.

B) Required packages installed (each node)

rpm -q kernel-devel kernel-headers gcc gcc-c++ cpp binutils elfutils-libelf-devel python38
Red flag: any “is not installed” line means stop and fix before proceeding.

C) Time sync & hostname resolution (each node)

timedatectl status
getent hosts node01 node02 node03
Why this matters: clusters depend on consistent time and consistent name resolution.

D) Disk visibility (each node)

lsblk -o NAME,SIZE,TYPE,MODEL
ls -l /dev/disk/by-id | head -n 50
Why this matters: you can’t build NSDs reliably if device naming changes every reboot.

E) Installer extraction proof (each node)

# Show the extracted version directory exists
ls -ld /usr/lpp/mmfs/*
ls -l /usr/lpp/mmfs/*/gpfs_rpms | head -n 40
Pass: you can prove the payload exists and contains gpfs rpms for your platform.

F) Cluster creation & health proof (run on the manager node)

mmlscluster
mmgetstate -aLs
mmgetstate -a
# Optional: show where mm binaries live
ls -l /usr/lpp/mmfs/bin | head -n 30
Instructor shortcut: if mmgetstate -aLs doesn’t show what you expect, don’t guess—paste the output.
